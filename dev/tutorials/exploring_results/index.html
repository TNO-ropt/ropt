    Exploring Results

    This tutorial shows how to inspect the information available after an optimization run. It covers three approaches: examining individual FunctionResults objects, browsing the full result history with HistoryHandler, and collecting results into pandas DataFrames with TableHandler.

    Source Code

    The complete source code for this tutorial is available at examples/exploring_results.py.

    Prerequisites

    This tutorial builds on the concepts from the Workflow Framework and Constrained Optimization tutorials. It uses a small stochastic Rosenbrock problem with a nonlinear constraint so that every result type has data to display.

    Setup

    The configuration uses 3 variables, 5 realizations, and a single nonlinear constraint with an upper bound of 1.0. We limit the run to 4 function evaluations to keep the output readable:

    import numpy as np
from numpy.random import default_rng

from ropt.context import EnOptContext
from ropt.enums import EnOptEventType
from ropt.evaluation import EvaluationBatchContext, EvaluationBatchResult
from ropt.events import EnOptEvent
from ropt.results import FunctionResults, GradientResults
from ropt.workflow.compute_steps import OptimizationStep
from ropt.workflow.evaluators import BatchEvaluator
from ropt.workflow.event_handlers import (
    CallbackHandler,
    HistoryHandler,
    ResultHandler,
    TableHandler,
)

DIM = 3
REALIZATIONS = 5
CONFIG = {
    "variables": {
        "variable_count": DIM,
        "perturbation_magnitudes": 1e-6,
    },
    "realizations": {
        "weights": [1.0] * REALIZATIONS,
    },
    "gradient": {
        "number_of_perturbations": 5,
    },
    "optimizer": {
        "max_functions": 4,
    },
    "nonlinear_constraints": {
        "lower_bounds": -np.inf,
        "upper_bounds": 1.0,
    },
}
INITIAL_VALUES = np.array([0.5, 1.0, 1.5])

    The evaluation callback computes the Rosenbrock objective together with a constraint \(x_1^2 + x_2^2\). It also assigns a batch_id to each result so that batches can be tracked across function and gradient evaluations:

    batch_counter = 0

def rosenbrock(variables, context, a, b):
    global batch_counter
    objectives = np.zeros((variables.shape[0], 1), dtype=np.float64)
    constraints = np.zeros((variables.shape[0], 1), dtype=np.float64)
    for v_idx, r in enumerate(context.realizations):
        for d_idx in range(DIM - 1):
            x, y = variables[v_idx, d_idx : d_idx + 2]
            objectives[v_idx, 0] += (a[r] - x) ** 2 + b[r] * (y - x * x) ** 2
        constraints[v_idx, 0] = variables[v_idx, 0] ** 2 + variables[v_idx, 1] ** 2
    batch_id = batch_counter
    batch_counter += 1
    return EvaluationBatchResult(
        objectives=objectives, constraints=constraints, batch_id=batch_id,
    )

    The batch_id field on EvaluationBatchResult is optional. When set, it propagates to the resulting FunctionResults and GradientResults objects and appears in table output.

    Running the Optimization

    We set up the workflow with three event handlers so that every piece of result data is captured:

    • ResultHandler — tracks the best result.
    • HistoryHandler — collects every result.
    • TableHandler — builds pandas DataFrames from results.
    rng = default_rng(seed=42)
a = rng.normal(loc=1.0, scale=0.1, size=REALIZATIONS)
b = rng.normal(loc=100.0, scale=10.0, size=REALIZATIONS)

evaluator = BatchEvaluator(callback=partial(rosenbrock, a=a, b=b))
step = OptimizationStep(evaluator=evaluator)

result_handler = ResultHandler()
step.add_event_handler(result_handler)

history = HistoryHandler()
step.add_event_handler(history)

table_handler = TableHandler()
table_handler.set_default_tables()
step.add_event_handler(table_handler)

context = EnOptContext.model_validate(CONFIG)
step.run(variables=INITIAL_VALUES, context=context)

    Part 1: Inspecting a Single Result

    After the run, result_handler["results"] holds the best FunctionResults found. This object contains four groups of information:

    Aggregated function values

    The functions field holds objective and constraint values aggregated across all realizations:

    result = result_handler["results"]
assert result.functions is not None

print(result.functions.target_objective)  # weighted sum used by the optimizer
print(result.functions.objectives)        # per-objective values, shape (n_objectives,)
print(result.functions.constraints)       # per-constraint values, shape (n_constraints,)

    target_objective is a 0-D array containing the single scalar the optimizer minimizes — a weighted sum of all objectives across all realizations. objectives and constraints give the individual values.

    Per-realization evaluations

    The evaluations field holds the raw data for each realization:

    print(result.evaluations.variables)    # shape (n_variables,)
print(result.evaluations.objectives)   # shape (n_realizations, n_objectives)
print(result.evaluations.constraints)  # shape (n_realizations, n_constraints)

    variables is the variable vector that was evaluated. objectives and constraints are two-dimensional: one row per realization, one column per objective or constraint.

    Realization metadata

    The realizations field tracks which realizations were active and which failed:

    print(result.realizations.active_realizations)  # boolean array, shape (n_realizations,)
print(result.realizations.failed_realizations)  # boolean array, shape (n_realizations,)

    Constraint info

    When constraints are present, the constraint_info field reports how close each constraint is to its bounds and whether it is violated:

    if result.constraint_info is not None:
    # Signed distance to bounds (negative = violated for lower, positive = violated for upper)
    print(result.constraint_info.nonlinear_upper)     # shape (n_constraints,)

    # Absolute violation magnitude (0 when satisfied)
    print(result.constraint_info.nonlinear_violation)  # shape (n_constraints,)

    Violation data is available for all three constraint types: bound_violation, linear_violation, and nonlinear_violation.

    Part 2: Browsing the Result History

    HistoryHandler stores every result received during the run. After optimization, history["results"] is a tuple of all Results objects:

    all_results = history["results"]
print(f"Total results collected: {len(all_results)}")

for item in all_results:
    if isinstance(item, FunctionResults) and item.functions is not None:
        print(
            f"  Batch {item.batch_id} (functions): "
            f"objective = {item.functions.target_objective}"
        )
    elif isinstance(item, GradientResults) and item.gradients is not None:
        print(
            f"  Batch {item.batch_id} (gradients): "
            f"target gradient norm = {np.linalg.norm(item.gradients.target_objective):.6f}"
        )

    This lets you plot convergence curves, compare iterations, or perform post-processing on the full optimization trajectory.

    Example output:

    Total results collected: 5
  Batch 0 (functions): objective = 77.67011594154808
  Batch 1 (gradients): target gradient norm = 179.011256
  Batch 2 (functions): objective = 8861514722.699753
  Batch 3 (functions): objective = 1116158.765447345
  Batch 4 (functions): objective = 255.82775361717682

    Note how function and gradient batches alternate: the optimizer evaluates functions at the current point (batch 0), then estimates gradients via perturbations (batch 1), and uses those gradients to propose a new point (batch 2), and so on.

    Inspecting gradient results

    GradientResults objects have a structure similar to function results, but contain gradient information instead:

    grad = gradient_results[0]
assert grad.gradients is not None

# Gradient of the weighted objective w.r.t. each variable, shape (n_variables,)
print(grad.gradients.target_objective)

# Per-objective gradients, shape (n_objectives, n_variables)
print(grad.gradients.objectives)

# Per-constraint gradients (if constraints exist), shape (n_constraints, n_variables)
print(grad.gradients.constraints)

    The evaluations field contains the perturbed variable and objective data used to estimate the gradients:

    # shape (n_realizations, n_perturbations, n_variables)
print(grad.evaluations.perturbed_variables.shape)

# shape (n_realizations, n_perturbations, n_objectives)
print(grad.evaluations.perturbed_objectives.shape)

    Part 3: Tables via TableHandler

    TableHandler collects results into pandas DataFrames. Calling set_default_tables() registers three standard tables for function results, and two for gradient results:

    Table name Contents
    "functions" Target objective, per-objective and per-constraint values, variables.
    "evaluations" Per-realization objectives, constraints, and variables.
    "constraints" Constraint differences and violations for all constraint types.
    "gradients" Target objective gradient and per-objective/constraint gradients.
    "perturbations" Per-realization perturbed variable, objective, and constraint data.

    Access a table by name:

    functions_table = table_handler["functions"]
print(functions_table.to_string())

    The OCD prefix stands for Output Constraint Difference (nonlinear constraints). BCD and ICD columns appear when bound or linear constraints are configured. The violation column shows the absolute magnitude of any violation (0 when the constraint is satisfied).

    The gradients table collects gradient data from perturbation-based evaluations:

print(gradients_table.to_string())

    Each row corresponds to one gradient evaluation batch. The Total-Gradient columns show the gradient of the weighted objective with respect to each variable, while Grad-objective and Grad-constraint columns give the per-objective and per-constraint gradients.

    Custom tables

    You can register custom tables with add_table() to select exactly which fields to include. See Optimization Workflows for the column specification format.

    Summary

    This tutorial demonstrated three ways to access optimization results:

    1. Direct inspection of a FunctionResults object — useful for checking the final optimum or understanding the structure of result data.

    2. History tracking with HistoryHandler — gives access to every result (both function and gradient evaluations) for convergence analysis or post-processing.

    3. Table building with TableHandler — produces pandas DataFrames for functions, evaluations, constraints, and gradients, ready for display, export, or further analysis.

    For a complete reference on result fields, axes, and DataFrame export options, see Working with Results.
